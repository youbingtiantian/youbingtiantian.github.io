由此可以知道「ref」是返回了一个基本数据类型/复杂数据类型的响应式数据
那么我们首先要做的就是编写一个 function 叫做「ref」

createRef / isRef

在这里创建了「ref」函数，内部返回了一个「createRef」函数，将接收当前需要被 ref 创建的值，「shallow」参数可以不管，源码中也是 false，在「createRef」函数中，我们首先要做的事情是判断当前值是否已经是一个 ref，如果是，那么就直接返回，「isRef」函数中，我们判断当前值是否存在并且当前值的「__v_isRef」是否为 true，双感叹号的作用是将这里转为 boolean 类型，没错，这就是沙雕 JavaScript

那么我们显然发现这个「__v_isRef」和「new RefImpl」这两个莫名其妙的东西，其实 RefImpl 类，就是 ref 内部的实现关键，让我们看看这个类

RefImpl

在这里 new RefImpl 时 传递进入的 value 先被赋值给私有属性_rawValue（初始值）,同时，我们用_value 接收一个三元表达式结果，在这里我们知道__v_isShallow 在我们传进来的时候是 false 所以会执行「toReacitve」方法去判断当前是普通数据类型 / 复杂数据类型，在此我们也清楚了「__v_isRef」是怎么来的，为什么可以判断当前是否是 ref 数据这一疑问

在这里我要说明，「ref」方法，本质上对于使用者传递复杂数据类型进入时，会将其在内部调用「reactive」方法进行「proxy」代理，所以本质上来说当传入复杂数据类型时，基本等于是「reactive」帮使用者做了拦截，从而达到目的

在下方我们定义了类的 get 和 set 方法，在 get 中我们调用了收集依赖，传入 this 本身，在其中我们进行了依赖的收集，将 dep 传入其中，如果没有 dep 则创建 dep 关于「activeEffect」、「trackEffects」请看上一章「reactive」

在 set 中，首先使用「hasChanged」方法接收新旧参数，使用 Object.is() 进行判断，如果不相等，我们将新值赋给原始值，然后进行依赖的触发和「reactive」中的做法差不多

个人总结

mac 不太会用，写的比较粗略潦草，贴图和文字顺序可能不太准确，凑合看吧。

其实简单来讲「ref」的做法是针对于普通数据类型和复杂数据类型的值做了两个逻辑处理，如果是普通数据类型，其实在使用者的调用过程中，是触发了 RefImpl 类的 get ，从而进行了依赖的收集触发，set 进行依赖的触发。如果是复杂数据类型则交给了「reactive / proxy」去处理，这里可以认为 ref 对普通数据类型是没有响应性的，只是通过代码在内部手动触发了依赖，在这里声明的 get 和 set （“访问器函数”）值为 value，这也是为什么在使用 ref 时，在 js 中要使用 ‘.value’的形式去获取/更新数据了～
